# nonestore 说明书

## 1. nonestore 是做什么的

`nonestore` 是一个面向 React 的轻量状态管理库。

它主要解决两个问题：

1. 当多个组件需要共享同一份数据时，状态应该放在哪里。
2. 当状态更新逻辑越来越多时，如何把“读状态”和“改状态”分开管理。

你可以把它想象成一个“全局数据仓库”。

- `state` 是仓库里存放的数据。
- `actions` 是仓库管理员，专门负责修改数据。
- `computed` 是报表系统，专门根据已有数据计算出新的结果。
- `useStore` / `createStore` 返回的 hook 是读取仓库数据的窗口。

它的特点是：

- 写法简单
- 直接支持 React
- 可以像直接改对象一样改状态
- 支持派生数据
- 支持中间件扩展
- 支持持久化

---

## 2. 为什么不用普通的 useState

`useState` 很适合组件内部的小状态，比如：

- 一个输入框的值
- 一个弹窗是否打开
- 一个按钮是否 loading

但是当出现下面这些场景时，单纯使用 `useState` 会逐渐变乱：

- 多个组件都要读同一份数据
- 多个地方都能修改同一份数据
- 需要把修改逻辑统一收口
- 需要记录日志、校验、持久化、撤销重做

例如电商项目中的购物车：

- 商品列表页可以加购物车
- 商品详情页可以加购物车
- 购物车页可以改数量
- 顶部导航要显示购物车总数

这时如果每个组件都自己维护一份状态，很快就会失控。

`nonestore` 的作用，就是把共享状态抽出来，统一管理。

---

## 3. 核心思想

`nonestore` 的核心思想可以总结成一句话：

> 把“状态是什么”和“状态怎么改”分开。

### 3.1 state

状态本身，例如：

```ts
{ count: 0, age: 18 }
```

### 3.2 actions

动作，也就是“如何修改状态”。

例如：

```ts
increment: (state) => {
  state.count += 1;
};
```

### 3.3 computed

派生值，也就是“根据已有状态计算出来的结果”。

例如：

```ts
doubled: (state) => state.count * 2;
```

这里 `doubled` 不是手动存进去的，而是由 `count` 计算出来的。

这和数学里的函数很像：

- 已知 `count = 2`
- 可以计算 `doubled = 4`

---

## 4. 最基础的使用方式

### 4.1 创建 store

```tsx
import { createStore } from "nonestore";

const [useCounter, counterStore] = createStore(
  { count: 0 },
  {
    computed: {
      doubled: (state) => state.count * 2,
    },
    actions: {
      increment: (state) => {
        state.count += 1;
      },
      add: (state, amount: number) => {
        state.count += amount;
      },
    },
  },
);
```

这个 `createStore(...)` 返回两个东西：

1. `useCounter`
   React hook，用来在组件中读取状态。
2. `counterStore`
   store 实例，用来调用 actions、获取状态、订阅状态。

## 5. 在 React 中怎么用

### 5.1 完整示例：计数器

```tsx
import { createStore } from "nonestore";

const [useCounter, counterStore] = createStore(
  { count: 0 },
  {
    computed: {
      doubled: (state) => state.count * 2,
    },
    actions: {
      increment: (state) => {
        state.count += 1;
      },
      add: (state, amount: number) => {
        state.count += amount;
      },
      reset: (state) => {
        state.count = 0;
      },
    },
  },
);

function App() {
  const count = useCounter((state) => state.count);
  const doubled = useCounter((state) => state.doubled);

  return (
    <div>
      <h1>Counter Demo</h1>
      <p>count: {count}</p>
      <p>doubled: {doubled}</p>

      <button onClick={() => counterStore.actions.increment()}>+1</button>
      <button onClick={() => counterStore.actions.add(5)}>+5</button>
      <button onClick={() => counterStore.actions.reset()}>reset</button>
    </div>
  );
}

export default App;
```

### 5.2 这段代码发生了什么

当点击 `+1` 按钮时：

1. 执行 `counterStore.actions.increment()`
2. `increment` 修改 `state.count`
3. store 内部状态更新
4. 使用了 `useCounter((state) => state.count)` 的组件会重新渲染
5. `computed.doubled` 会根据最新的 `count` 重新计算

## 6. createStore 的结构怎么理解

```tsx
const [useCounter, counterStore] = createStore(initialState, config);
```

### 第一个参数：`initialState`

表示初始状态。

```ts
{
  count: 0;
}
```

### 第二个参数：`config`

是配置对象，目前最常用的是：

- `computed`
- `actions`
- `middlewares`
- `persist`
- `devtools`

## 7. actions 是什么

`actions` 用来定义“改状态的方法”。

### 7.1 形象理解

如果把 store 想成银行账户：

- `state.balance` 是余额
- `deposit` 是存钱动作
- `withdraw` 是取钱动作

动作本身不负责展示，它只负责修改数据。

### 7.2 示例

```tsx
actions: {
  increment: (state) => {
    state.count += 1
  },
  add: (state, amount: number) => {
    state.count += amount
  },
}
```

### 7.3 特点

- 第一个参数 `state` 就是当前状态
- 可以直接写 `state.count += 1`
- 可以接收额外参数
- 调用时统一走 `store.actions.xxx()`

例如：

```tsx
counterStore.actions.increment();
counterStore.actions.add(10);
```

### 7.4 常见错误

错误写法：

```tsx
const [useNameAge, nameageStore] = createStore(
  { age: 18 },
  {
    add: {
      addone: (state) => {
        state.age += 1;
      },
    },
  },
);
```

这里把动作写到 `add` 下面了，这是错的。

正确写法：

```tsx
const [useNameAge, nameageStore] = createStore(
  { age: 18 },
  {
    actions: {
      addone: (state) => {
        state.age += 1;
      },
      deleteone: (state) => {
        state.age -= 1;
      },
    },
  },
);
```

调用时也应该写：

```tsx
nameageStore.actions.addone();
nameageStore.actions.deleteone();
```

而不是：

```tsx
nameageStore.add.addone();
```

## 8. computed 是什么

`computed` 用来定义“派生值”。

### 8.1 形象理解

如果 `state` 是原材料，那么 `computed` 就像加工后的结果。

例如：

- 原材料：单价、数量
- 派生值：总价

```ts
price = 10;
qty = 3;
total = 30;
```

这里 `total` 不需要手动存进去，它只要根据 `price` 和 `qty` 算出来就行。

### 8.2 示例

```tsx
computed: {
  doubled: (state) => state.count * 2,
  isAdult: (state) => state.age >= 18,
}
```

### 8.3 使用方式

```tsx
const doubled = useCounter((state) => state.doubled);
const isAdult = useUser((state) => state.isAdult);
```

### 8.4 computed 和 actions 的区别

- `actions` 负责修改状态
- `computed` 负责根据状态计算结果

一句话：

> `actions` 是“做事的人”，`computed` 是“算结果的人”。

---

## 9. 选择器 selector 是什么

当你写：

```tsx
const count = useCounter((state) => state.count);
```

括号里的 `(state) => state.count` 就是 selector，叫“选择器”。

它的作用是：

> 告诉 store：我只关心 `count`。

### 为什么 selector 很重要

如果一个状态对象很大：

```ts
{
  count: 1,
  user: { name: 'Tom' },
  theme: 'dark'
}
```

而你的组件只需要 `count`，那就没必要在 `theme` 改变时也重新渲染。

所以 selector 的价值是：

- 精准读取数据
- 减少不必要渲染
- 让组件职责更清晰

## 10. 一个更贴近业务的 React 案例

### 10.1 用户资料案例

```tsx
import { createStore } from "nonestore";

const [useUserStore, userStore] = createStore(
  {
    name: "Alice",
    age: 17,
  },
  {
    computed: {
      isAdult: (state) => state.age >= 18,
      profileText: (state) => `${state.name} (${state.age})`,
    },
    actions: {
      setName: (state, name: string) => {
        state.name = name;
      },
      growUp: (state) => {
        state.age += 1;
      },
    },
  },
);

function UserCard() {
  const profileText = useUserStore((state) => state.profileText);
  const isAdult = useUserStore((state) => state.isAdult);

  return (
    <div>
      <p>{profileText}</p>
      <p>{isAdult ? "成年人" : "未成年人"}</p>
      <button onClick={() => userStore.actions.growUp()}>长大一岁</button>
      <button onClick={() => userStore.actions.setName("Bob")}>改名</button>
    </div>
  );
}
```

### 10.2 教学重点

这个案例很适合讲以下概念：

- `name` 和 `age` 是原始状态
- `isAdult` 和 `profileText` 是派生值
- `growUp` 和 `setName` 是修改动作
- 组件通过 hook 读取数据，通过 store 的 actions 修改数据

## 11. create 和 createStore 的区别

### `createStore`

这是 React 项目里最常用的入口。

```tsx
const [useStore, store] = createStore(initialState, config);
```

适合：

- React 组件中直接使用
- 教学示例
- 页面业务状态管理

### `create`

这是框架无关版本。

```ts
import { create } from "nonestore";

const store = create(
  { count: 0 },
  {
    actions: {
      increment: (state) => {
        state.count += 1;
      },
    },
  },
);
```

适合：

- 非 React 场景
- 工具函数里读取状态
- 服务层逻辑
- 单元测试

---

## 12. store 实例上有哪些东西

通过 `createStore` 返回的第二个值，就是 store 实例。

例如：

```tsx
const [useCounter, counterStore] = createStore(...)
```

这里 `counterStore` 上常见的能力有：

### 12.1 `counterStore.getState()`

读取当前完整状态。

```tsx
console.log(counterStore.getState());
```

### 12.2 `counterStore.setState(...)`

直接修改状态。

```tsx
counterStore.setState((state) => {
  state.count = 100;
});
```

也可以传对象：

```tsx
counterStore.setState({ count: 100 });
```

### 12.3 `counterStore.actions.xxx()`

调用动作。

```tsx
counterStore.actions.increment();
```

### 12.4 `counterStore.subscribe(...)`

订阅状态变化。

```tsx
const unsubscribe = counterStore.subscribe((state, prevState) => {
  console.log("new state:", state);
  console.log("prev state:", prevState);
});
```

取消订阅：

```tsx
unsubscribe();
```

### 12.5 `counterStore.destroy()`

清理 store。

一般日常 React 页面中不常主动使用，但某些特殊场景下会用到。

## 13. 持久化 persist

如果你希望刷新页面后，某些状态还能保留，可以使用 `persist`。

### 示例

```tsx
import { createStore } from "nonestore";

const [useSettings, settingsStore] = createStore(
  {
    theme: "light",
    language: "zh-CN",
    tempText: "",
  },
  {
    persist: {
      key: "settings-store",
      pick: ["theme", "language"],
    },
  },
);
```

### 形象理解

可以把 `persist` 理解成“自动存档”。

- 页面关闭前不用你手动保存
- 下次打开时会自动读取之前的记录

### `pick` 有什么用

`pick` 表示：

> 只持久化这些字段。

比如临时输入框内容 `tempText` 不重要，就不需要长期保存。

## 14. 中间件 middlewares

中间件是对状态更新过程的扩展。

可以把它理解成“状态更新流水线上的附加处理器”。

例如：

- 打印日志
- 做合法性校验
- 记录历史快照
- 批量更新

### 使用方式

```tsx
import { createStore, logger, validate } from "nonestore";

const [useCounter] = createStore(
  { count: 0 },
  {
    middlewares: [
      logger({ collapsed: true, diff: true }),
      validate((state) => {
        if (state.count < 0) return "count 不能小于 0";
      }),
    ],
  },
);
```

---

## 15. 内置中间件说明

### 15.1 logger

作用：在控制台打印状态变化。

适合教学时观察：

- 更新前是什么
- 更新后是什么
- 改了哪些字段

示例：

```tsx
logger({ collapsed: true, diff: true });
```

### 15.2 validate

作用：每次状态变化后做校验。

示例：

```tsx
validate((state) => {
  if (state.count < 0) return "count 不能小于 0";
});
```

如果返回字符串，就会抛出错误。

### 15.3 timeTravel

作用：记录历史状态，实现撤销和重做。

示例：

```tsx
import { createStore, timeTravel } from "nonestore";

const travel = timeTravel<{ count: number }>();

const [useCounter] = createStore(
  { count: 0 },
  {
    middlewares: [travel.middleware],
  },
);

travel.undo();
travel.redo();
console.log(travel.history());
```

### 15.4 batch

作用：把短时间内的多次更新合并处理。

教学中可以这样理解：

> 连续敲很多次命令，不要每敲一次都立刻结算，而是等本轮结束后一起处理。

### 15.5 dedup

当前版本导出了 `dedup()`，但目前实现比较基础。教学时可以把它介绍为“用于未来避免重复更新的扩展点”，不要夸大它的实际能力。

## 16. async action

`actions` 也可以写成异步函数。

例如：

```tsx
actions: {
  fetchUser: async (state, id: number) => {
    const res = await fetch(`/api/user/${id}`);
    const data = await res.json();
    state.user = data;
  };
}
```

如果初始状态中有以下字段：

- `fetchUserLoading`
- `fetchUserError`

那么在执行这个异步 action 时，store 会尝试管理这两个字段。

例如：

```tsx
const [useUser, userStore] = createStore(
  {
    user: null as null | { name: string },
    fetchUserLoading: false,
    fetchUserError: null as unknown,
  },
  {
    actions: {
      fetchUser: async (state, id: number) => {
        const res = await fetch(`/api/user/${id}`);
        state.user = await res.json();
      },
    },
  },
);
```

这是一个比较适合课堂演示“异步状态”的例子。

## 17. 为什么可以直接写 state.count += 1

很多学生会问：

> React 里不是强调不能直接改对象吗？为什么这里可以直接改？

答案是：

`nonestore` 内部使用了 Immer。

Immer 的作用是：

- 让你看起来像是在“直接修改”
- 实际上底层会生成一个新的不可变状态

所以你可以写：

```tsx
state.count += 1;
```

而不是每次都手写：

```tsx
return { ...state, count: state.count + 1 };
```

这会让代码更容易教，也更容易让初学者接受。

## 18. 教学时最值得强调的几个规则

### 规则 1：共享数据放进 store

如果多个组件都要用，就不要每个组件自己维护一份。

### 规则 2：修改数据走 actions

不要在组件里到处散落复杂修改逻辑。

### 规则 3：能算出来的数据就放 computed

不要重复存储“可以被推导出来的数据”。

例如：

- 应该存 `price` 和 `qty`
- 不一定要手动存 `total`
- 因为 `total` 可以由 `computed` 算出来

### 规则 4：组件里优先用 selector 精准读取

```tsx
const count = useCounter((state) => state.count);
```

比：

```tsx
const state = useCounter();
```

更清晰，也更利于性能。

## 19. 学生最容易犯的错误

### 错误 1：把 actions 写错位置

错误：

```tsx
{
  add: {
    addone: () => {};
  }
}
```

正确：

```tsx
{
  actions: {
    addone: () => {};
  }
}
```

### 错误 2：调用 action 时漏掉 actions

错误：

```tsx
store.addone();
store.add.addone();
```

正确：

```tsx
store.actions.addone();
```

### 错误 3：把 computed 当成可修改状态

`computed` 是算出来的，不应该直接赋值。

### 错误 4：把本地 UI 小状态也全部塞进全局 store

不是所有状态都值得放全局。

例如一个输入框是否聚焦，这种通常还是 `useState` 更合适。

## 20. 一套适合课堂演示的步骤

你可以按下面顺序讲：

1. 先讲什么是共享状态
2. 再讲 `createStore` 的返回值
3. 再讲 `actions`：如何修改状态
4. 再讲 `computed`：如何派生数据
5. 再讲 selector：为什么只取自己需要的字段
6. 再演示 `persist`
7. 最后再扩展到 `logger`、`validate`、`timeTravel`

这样学生比较容易形成完整理解。

---

## 21. 课堂总结

`nonestore` 最重要的不是 API 数量，而是它背后的设计思想：

- 状态集中管理
- 修改逻辑统一收口
- 派生数据不要重复存储
- 组件按需订阅数据

如果学生把这四点真正理解了，那么他们以后学 Zustand、Redux、Jotai，都会更容易。

---

## 22. 一句话总结

`nonestore` 可以把 React 中分散、重复、难维护的共享状态，整理成：

- 清晰的状态
- 清晰的动作
- 清晰的派生值
- 清晰的使用方式

这就是它最适合教学的地方。
