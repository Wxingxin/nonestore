# 在中等项目中如何使用 nonestore

## 1. 先回答核心问题

在中等项目中使用 `nonestore`，**建议把 store 提取到专门的文件夹中集中管理**，但这里的“集中”不是把所有状态都塞进一个巨大文件里，而是：

- 按业务模块集中管理
- 按目录规范统一管理
- 让状态逻辑从组件中抽离出来
- 让团队能快速找到每个 store 在哪里

所以更准确的说法不是：

> 所有状态写进一个文件。

而是：

> 把共享状态统一放进一个专门的状态目录里，再按业务模块拆分。

这才是中等项目里更合理的做法。

---

## 2. 为什么中等项目不能继续随便写

小项目里，状态可能只有几个：

- 计数器
- 用户信息
- 主题切换

这种时候，把 store 直接写在页面文件里问题不大。

但到了中等项目，通常会出现：

- 页面数量变多
- 多个模块同时存在
- 多个组件共享同一份状态
- 多人协作
- 状态修改逻辑越来越多

如果这时还把 store 直接散落在页面里，就会出现几个问题：

1. 找不到状态定义在哪
2. 组件文件越来越大
3. 修改逻辑分散，难维护
4. 多人改代码时容易冲突
5. 状态边界越来越模糊

所以到了中等项目，store 应该从页面组件中“搬出来”。

---

## 3. 中等项目的基本原则

在中等项目里使用 `nonestore`，建议遵守下面几个原则。

### 原则 1：共享状态放 store，本地 UI 状态留在组件里

不是所有状态都要放进 `nonestore`。

适合放 store 的：

- 用户信息
- 购物车
- 筛选条件
- 列表查询参数
- 权限信息
- 主题设置
- 多组件共享的业务数据

不一定要放 store 的：

- 一个弹窗是否打开
- 一个输入框是否聚焦
- 某个按钮 hover 状态
- 单个组件内部的小交互状态

一句话：

> 只有“需要共享”或“需要统一管理”的状态，才值得放进 store。

---

### 原则 2：按业务模块拆 store，不要做一个全局超级 store

中等项目最容易犯的错误是：

> 做一个超级大的 `appStore`，所有东西都往里面塞。

这样短期看似方便，长期一定会出问题：

- 状态对象越来越大
- action 越来越多
- 模块边界越来越模糊
- 任意页面都可能修改任意状态
- 维护难度快速上升

更好的方式是按业务模块拆分。

例如：

- `user` 模块一个 store
- `cart` 模块一个 store
- `settings` 模块一个 store
- `productFilter` 模块一个 store
- `dashboard` 模块一个 store

也就是说：

> 一个 store 只负责一类业务问题。

---

### 原则 3：组件只负责展示和触发 action，业务逻辑放到 store 中

组件中应该尽量少写复杂的状态处理逻辑。

组件里更适合做的是：

- 读数据
- 触发 action
- 渲染 UI

例如：

```tsx
const count = useCounterStore((state) => state.count)
<button onClick={() => counterStore.actions.increment()}>+1</button>
```

而不是在组件里到处写一堆状态拼接和计算逻辑。

这能让：

- 组件更干净
- store 更像业务状态中心
- 逻辑更容易复用

---

### 原则 4：computed 只放派生值，不要滥用

`computed` 应该只做“由已有状态推导出来的值”。

例如：

- 总价
- 是否成年
- 过滤后的结果数
- 用户展示文案

不要把它变成“什么逻辑都往里塞”的地方。

中等项目里如果 `computed` 滥用，会让依赖关系越来越难看懂。

---

## 4. 推荐目录结构

在中等项目中，建议单独创建一个 `stores` 目录，专门放 `nonestore` 相关代码。

例如：

```txt
src/
├─ components/
├─ pages/
├─ services/
├─ stores/
│  ├─ user/
│  │  ├─ index.ts
│  │  ├─ types.ts
│  │  └─ selectors.ts
│  ├─ cart/
│  │  ├─ index.ts
│  │  ├─ types.ts
│  │  └─ selectors.ts
│  ├─ settings/
│  │  ├─ index.ts
│  │  ├─ types.ts
│  │  └─ selectors.ts
│  └─ index.ts
├─ utils/
└─ App.tsx
```

这是一种比较适合中等项目的结构。

### 为什么这样组织

好处有几个：

1. 所有人都知道共享状态在 `stores/` 目录里
2. 每个业务模块都有自己的 store 文件夹
3. store 的类型、selector 可以顺手拆开
4. 后面项目继续长大时，目录不会立刻失控

---

## 5. 一个更贴近实际的组织方式

下面给一个中等项目更实用的例子。

### 5.1 用户模块

`src/stores/user/index.ts`

```ts
import { createStore } from 'nonestore'

export type UserState = {
  id: number | null
  name: string
  role: 'guest' | 'admin' | 'editor'
}

export const [useUserStore, userStore] = createStore(
  {
    id: null,
    name: '',
    role: 'guest' as const,
  },
  {
    computed: {
      isLogin: (state) => state.id !== null,
      isAdmin: (state) => state.role === 'admin',
    },
    actions: {
      login: (state, payload: { id: number; name: string; role: 'admin' | 'editor' }) => {
        state.id = payload.id
        state.name = payload.name
        state.role = payload.role
      },
      logout: (state) => {
        state.id = null
        state.name = ''
        state.role = 'guest'
      },
    },
  }
)
```

这个 store 的职责很清楚：

- 只负责用户信息
- 不负责购物车
- 不负责主题设置
- 不负责页面临时弹窗状态

这就是“按业务边界拆分”。

---

### 5.2 购物车模块

`src/stores/cart/index.ts`

```ts
import { createStore } from 'nonestore'

export type CartItem = {
  id: number
  name: string
  price: number
  qty: number
}

export const [useCartStore, cartStore] = createStore(
  {
    items: [] as CartItem[],
  },
  {
    computed: {
      total: (state) => state.items.reduce((sum, item) => sum + item.price * item.qty, 0),
      itemCount: (state) => state.items.reduce((sum, item) => sum + item.qty, 0),
    },
    actions: {
      addItem: (state, item: CartItem) => {
        const existing = state.items.find((it) => it.id === item.id)
        if (existing) {
          existing.qty += item.qty
        } else {
          state.items.push(item)
        }
      },
      removeItem: (state, id: number) => {
        state.items = state.items.filter((item) => item.id !== id)
      },
      clearCart: (state) => {
        state.items = []
      },
    },
  }
)
```

这个 store 只做购物车，不碰用户信息。

这样模块边界就比较清楚。

---

### 5.3 设置模块

`src/stores/settings/index.ts`

```ts
import { createStore } from 'nonestore'

export const [useSettingsStore, settingsStore] = createStore(
  {
    theme: 'light',
    language: 'zh-CN',
  },
  {
    persist: {
      key: 'settings-store',
      pick: ['theme', 'language'],
    },
    actions: {
      setTheme: (state, theme: string) => {
        state.theme = theme
      },
      setLanguage: (state, language: string) => {
        state.language = language
      },
    },
  }
)
```

这个 store 比较适合配合 `persist` 使用，因为它本身就是设置类状态。

## 6. 页面组件中应该怎么使用

在中等项目里，组件中只做“读取 + 调用”，不要把 store 本身写回组件里反复定义。

例如：

```tsx
import { useCartStore, cartStore } from '@/stores/cart'

function CartPanel() {
  const items = useCartStore((state) => state.items)
  const total = useCartStore((state) => state.total)

  return (
    <div>
      <p>items: {items.length}</p>
      <p>total: {total}</p>
      <button onClick={() => cartStore.actions.clearCart()}>clear</button>
    </div>
  )
}
```

这类页面组件的特点应该是：

- 不关心 store 怎么创建
- 只关心如何读取和使用
- 尽量保持视图层干净

---

## 7. 中等项目里要不要做一个 `stores/index.ts`

可以做，但作用应该只是“统一导出”，不是“统一揉成一个超级 store”。

例如：

`src/stores/index.ts`

```ts
export * from './user'
export * from './cart'
export * from './settings'
```

这样做的优点：

- 导入路径更统一
- 对外暴露更清晰
- 后面重构目录时影响更小

但要注意：

> `stores/index.ts` 只是导出入口，不是把所有状态混在一起。

---

## 8. selector 在中等项目里为什么更重要

中等项目状态变多后，selector 的价值会更明显。

例如：

```tsx
const total = useCartStore((state) => state.total)
```

而不是：

```tsx
const cart = useCartStore()
```

原因很简单：

- 代码更清晰
- 组件依赖更明确
- 避免不必要的重新渲染
- 方便后续维护

在中等项目里，建议形成团队习惯：

> 优先使用精确 selector，而不是整块读取 state。

---

## 9. 是否需要给每个 store 单独写 types.ts

中等项目里，建议这样判断：

### 情况 1：状态很简单

例如：

- 只有 2 到 3 个字段
- action 不复杂

那可以直接写在 `index.ts` 里。

### 情况 2：状态和 action 越来越复杂

就建议拆出 `types.ts`。

例如：

```txt
stores/
└─ cart/
   ├─ index.ts
   ├─ types.ts
   └─ selectors.ts
```

适合拆类型的情况：

- 状态字段多
- 业务模型复杂
- 需要多个文件复用类型
- 需要和接口类型联动

---

## 10. 是否要给每个 store 单独写 selectors.ts

不是必须，但中等项目里是个好习惯。

例如：

`src/stores/cart/selectors.ts`

```ts
export const selectCartItems = (state: { items: any[] }) => state.items
export const selectCartTotal = (state: { total: number }) => state.total
```

然后组件里：

```tsx
const total = useCartStore(selectCartTotal)
```

这样做的好处：

- selector 可复用
- 命名更清晰
- 大家不需要每次都现场写匿名函数
- 更适合多人协作

但中等项目初期也不必强制全拆，避免过度工程化。

---

## 11. 中等项目中推荐的管理方式

如果项目规模属于中等，推荐采用下面这种模式。

### 模式：模块化 store 管理

规则如下：

1. 所有共享状态都放在 `src/stores/`
2. 一个业务模块一个 store 文件夹
3. 每个 store 只负责一个明确业务域
4. 组件只读状态和触发 action
5. 尽量用 selector 精确读取
6. 设置类状态优先考虑 `persist`
7. 关键 store 可以加 `logger` 或 `validate`

这是一种“够用、清晰、容易执行”的中项目方案。

---

## 12. 不推荐的用法

### 不推荐 1：所有状态都写进一个 `appStore`

坏处：

- 文件膨胀
- 边界混乱
- 团队协作困难

### 不推荐 2：每个页面自己偷偷创建一份共享 store

坏处：

- 重复定义
- 状态难统一
- 找不到状态入口

### 不推荐 3：把局部 UI 状态也全部塞进 store

坏处：

- 增加复杂度
- 无意义抽象
- 组件本地状态反而更难写

### 不推荐 4：组件里直接写大量衍生逻辑

如果很多组件都在重复写：

```tsx
const total = items.reduce(...)
```

那就应该考虑是不是应该放进 `computed`。

## 14. 中等项目里 nonestore 的优点

如果按上面这种方式组织，`nonestore` 在中等项目里有这些优点：

1. 比散乱的 `useState` 更清晰
2. 比重型状态方案更轻量
3. 比直接把逻辑写在组件里更容易维护
4. 组件与业务状态职责更清楚
5. 新成员能快速找到共享状态在哪

也就是说：

> 对中等项目来说，只要组织方式对，`nonestore` 是够用的。

---

## 15. 中等项目里它的边界在哪里

虽然能用，但也要知道边界。

到了中等项目后期，如果出现下面这些情况，就要更谨慎：

- store 之间开始大量互相依赖
- 异步状态越来越复杂
- 服务端缓存逻辑越来越多
- 团队人数快速增加
- 调试和追踪需求大幅上升

这时候就要考虑：

- 是否继续拆模块
- 是否把服务端状态交给专门的请求库
- 是否补充更强的工程约束

所以 `nonestore` 在中等项目里是“能用”，但前提是你要有组织能力。

---

## 16. 最后的建议

如果你的项目是中等规模，最推荐的做法是：

### 推荐做法

- 建立 `stores/` 目录
- 按业务模块拆 store
- 每个 store 单独负责一个问题域
- 不做超级全局 store
- 组件只读和调用 action
- 尽量使用 selector
- 设置类状态用 `persist`
- 必要时补 `types.ts` 和 `selectors.ts`

### 一句话总结

> 在中等项目里，`nonestore` 不是“所有状态塞进一个文件集中使用”，而是“把共享状态统一收进 `stores/` 目录，再按业务模块拆分管理”。

这才是更合理、更能长期维护的使用方式。
